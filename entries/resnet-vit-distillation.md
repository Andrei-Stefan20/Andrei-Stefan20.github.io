![Original Qwen architecture compared with the ResNet-50 and adapter pipeline](/images/projects/resnet-vit/hero.png)

Vision-Language Models are usually built around a fairly fixed idea. An image enters a Vision Transformer, becomes a sequence of embeddings and is then passed to the language model.

It works, so there is rarely a reason to question it.

I questioned it anyway.

The idea behind this project was to understand whether Qwen's visual encoder could be replaced with ResNet-50 without retraining the language model from scratch. Not by changing what Qwen expects to receive, but by teaching a completely different encoder to produce a compatible representation.

That difference is where the entire project lives.

## Replacing the encoder was not the real problem

Removing the original Vision Transformer and inserting ResNet-50 is relatively easy. The difficult part begins immediately afterwards.

A Vision Transformer produces a flat sequence of visual tokens. ResNet-50 produces hierarchical feature maps at different resolutions and depths. One architecture represents the image as a sequence, while the other gradually builds spatial features from local patterns.

The outputs have different shapes, different structures and different meanings.

Passing raw ResNet features directly to Qwen would be like replacing a translator with someone who speaks a completely different language and expecting the conversation to continue as before.

The real task was therefore not replacing the encoder. It was building the translator between them.

## Teaching ResNet to speak Qwen

The language model remains unchanged. ResNet-50 becomes the new visual backbone, while a learned adapter translates its intermediate feature maps into the representation Qwen already knows how to use.

![Multi-stage adapter used to connect ResNet-50 with Qwen](/images/projects/resnet-vit/adapter.png)

From Qwen's point of view, the interface stays the same. It still receives a sequence of 196 visual embeddings with dimensionality 2048. The difference is that those embeddings are reconstructed from CNN features instead of being generated by the original Vision Transformer.

This makes the adapter the most important part of the system. It is not learning to classify images. It is learning to imitate the internal visual representation of another model.

## The teacher was already inside Qwen

To train the new pipeline, I first used Qwen's original Vision Transformer as a teacher.

Each image was processed by the original encoder and the resulting embeddings were saved. The same image was then passed through ResNet-50, and the adapter tried to reconstruct the teacher embeddings from the CNN feature maps.

Training minimized the Mean Squared Error between the original representation and the reconstructed one.

In practice, the model was not directly learning that an image contained a sheep, a tank or a castle. It was learning how Qwen represents those images before generating a single word.

This is what made the problem interesting to me. Everything happens inside the latent space, before the final answer is visible.

## One adapter was not enough

The first version used a relatively simple convolutional fusion block. It gave us a baseline and showed that the mapping was possible.

The second introduced a Transformer Encoder after feature fusion to recover some of the global interactions naturally present in a ViT.

The third focused more on local spatial information by improving the convolutional fusion stage.

The fourth combined both ideas, using stronger local fusion together with Transformer-based global reasoning.

![Detailed model structure and tensor dimensions](/images/projects/resnet-vit/model-summary.png)

This iterative part was more useful than jumping directly to one large architecture. Each version made it easier to understand whether the model needed better local features, better global context or a balance between both.

## Respecting the interface

Qwen expects a tensor with shape `[B, 196, 2048]`. Producing something vaguely similar is not enough. The adapter must reach the exact token structure expected by the rest of the model.

ResNet feature maps begin with very different resolutions and channel dimensions. The adapter projects them into compatible spaces, fuses information from multiple stages, reduces the spatial dimensions and finally produces the required token sequence.

This strict interface kept the experiment focused. I did not redesign Qwen around ResNet. ResNet and the adapter had to fit the model that already existed.

## Testing images that have almost nothing in common

I used visually different examples because they make failure easier to notice.

![Tank used as an inference example](/images/projects/resnet-vit/tank.jpg)

The tank contains mechanical details, straight edges, text, tracks and a relatively cluttered background.

![Sheep used as an inference example](/images/projects/resnet-vit/sheep.jpg)

The sheep image is almost the opposite. It contains repeated subjects, softer shapes, texture and a stronger relationship between foreground and background.

![Castel del Monte used as an inference example](/images/projects/resnet-vit/castle.jpg)

Castel del Monte introduces symmetry, repeated geometric elements and a much wider scene.

These images are not a benchmark by themselves. They are useful because an encoder meant to replace a general-purpose ViT should not work only on one visual domain.

## Fine-tuning the full visual path

The first training stage kept ResNet frozen and optimized only the adapter. This forced the translation layer to work with stable CNN features.

After the adapter had learned a reasonable mapping, I also experimented with joint fine-tuning. ResNet and the adapter were trained together using different learning rates, followed by another adapter-only phase.

This allowed the backbone to move towards representations that were easier to translate without immediately destroying the useful features learned from ImageNet.

The process was slower and more delicate, but it turned the experiment into something more than a translation layer placed on top of a frozen network. The entire visual path started adapting to the representation expected by Qwen.

## What I took away from the project

Before this project, I mostly thought of the Vision Transformer as the component that allowed Qwen to understand images.

After working on it, I started thinking much more about the interface between the visual and linguistic parts.

The language model never sees an image. It sees tensors.

Matching the tensor shape is not enough, of course. Two tensors can have identical dimensions and represent completely different things. The difficult part is preserving the latent structure the language model has learned to use.

For that reason, the project became less about replacing a Transformer with a CNN and more about translating between two representation spaces.

I do not consider the current implementation a final answer. It is an experiment that makes the real difficulty visible and provides a pipeline for continuing the work. Cosine-based objectives, contrastive alignment, lighter adapters and more systematic evaluation are all directions worth exploring.

The central idea, however, remained valid. A multimodal model can be seen as a collection of components connected by learned interfaces. Once one of those interfaces is understood, parts that initially appear fixed become open to experimentation.
